# Running the VAE-NLME reference implementation (Rohleff et al. 2025)

How to get <https://github.com/janrohleff/vae_nlme> running on a modern Linux
box, so `est = "vae"` can be compared against it directly rather than by reading
its source.  Written against a checkout at `~/src/vae_nlme`.

Two obstacles, both solvable without the commercial solver.

## 1. The compiled extensions target the wrong Python

`VAE/encoder` and `ParaUpdate/pop_parameter` ship only as
`cpython-36m-x86_64-linux-gnu.so`, `cpython-313-darwin.so` and a Windows `.pyd`.
On a Python 3.12 interpreter every one of those fails to import.

Both were cythonized from plain `.py` files in pure-Python mode (see the Cython
Metadata block at the top of each `.c`), and **both `.c` files are in the repo**,
generated by Cython 3.1.2.  So they can just be compiled for the local
interpreter -- no Cython, no `.pyx`, no upstream changes:

```sh
cd ~/src/vae_nlme
for m in ParaUpdate/pop_parameter VAE/encoder; do
  gcc -shared -fPIC -O2 -I/usr/include/python3.12 "$m.c" \
      -o "${m}.cpython-312-x86_64-linux-gnu.so"
done
```

Needs the Python development headers (`/usr/include/python3.12/Python.h`).  The
new `.so` names do not collide with the shipped ones, so this is additive.

## 2. GUROBI is commercial

The covariate M-step calls `prob.solve(solver=cp.GUROBI)`.  The compiled module
resolves `cp.GUROBI` as an attribute lookup **at solve time**, so redirecting it
before the run is enough -- their code is untouched:

```python
import cvxpy as cp
cp.GUROBI = cp.SCIP     # free, MIQP-capable
```

`pip install cvxpy pyscipopt` provides SCIP.  Both solve the same MIQP to
optimality, so this is a backend swap, not an approximation.  (`cvxpy`'s other
free backends are MILP-only and cannot express the quadratic objective.)

## 3. Environment

The system Python may lack `ensurepip`, which makes `python3 -m venv` fail.
Bootstrapping pip avoids needing `sudo apt install python3.12-venv`:

```sh
python3 -m venv --without-pip ~/.venvs/vae
curl -sSLo get-pip.py https://bootstrap.pypa.io/get-pip.py
~/.venvs/vae/bin/python get-pip.py
~/.venvs/vae/bin/pip install torch torchode numpy cvxpy pyscipopt matplotlib scipy
```

`gurobipy` is deliberately omitted.  Use a headless matplotlib backend
(`matplotlib.use("Agg")`) or the plotting at the end of each case study blocks.

## What this makes possible

- **Decoder validation.**  Their `VAE/decoder.py` is pure Python and needs only
  `torch`/`torchode` -- no extensions, no solver.  Importing
  `Decoder_neonates` and rebuilding their ODE settings (Dopri5,
  `atol = rtol = 1e-9`) reproduces their neonatal predictions; `rxode2` agrees to
  1.0e-08 relative over all 1120 observations of the 189-subject dataset, using
  their own data and their own padded `t_eval` construction.  That is the solver
  tolerance, i.e. agreement to numerical precision.
- **Full pipeline.**  With both extensions rebuilt and the solver redirected,
  `Main/neonates.py` runs start to finish, so the reference's own selected
  covariate set can be read off directly instead of inferred from the paper's
  figures.

## Caveat

The encoder is already validated against torch independently
(`tests/testthat/test-vae-encoder.R` checks forward and analytic backward against
a torch autograd fixture to 1e-5), so a full-pipeline run is not needed to trust
our LSTM.  What a full run settles is the *composed* behavior over a training
trajectory, which per-component agreement cannot establish.
